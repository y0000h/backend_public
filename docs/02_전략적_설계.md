# 전략적 설계 (Strategic Design)

## 1. 개요
DDD의 상위 개념으로, **비즈니스 전반을 구조적으로 분리**하는 접근이다.

## 2. 주요 개념
### 🔹 하위 도메인 (Subdomain)
비즈니스 전체를 작은 문제영역으로 나눔.
- Core (핵심)
- Supporting (보조)
- Generic (공통)

### 🔹 바운디드 컨텍스트 (Bounded Context)
모델과 언어가 일관되게 유지되는 논리적 경계.
- 동일한 용어라도 다른 컨텍스트에서는 다른 의미를 가질 수 있음.

### 🔹 보편 언어 (Ubiquitous Language)
도메인 전문가와 개발자가 공유하는 공통 언어.  
모델, 클래스, API 이름이 모두 비즈니스 용어로 일관되어야 함.

## 3. 컨텍스트 간 관계 (Context Map)
| 관계 | 설명 |
|------|------|
| Partnership | 양측이 긴밀히 협력 |
| Shared Kernel | 공통 코드 일부 공유 |
| Anti-Corruption Layer | 외부 모델로부터 내부 모델 보호 |

## 4. 장단점
- ✅ 팀 단위 개발 병렬성 향상  
- ✅ 모델 충돌 최소화  
- ⚠️ 초기 컨텍스트 정의 어려움

## 5. 예시 개념 매핑
가령 B2B 전자상거래 플랫폼을 예로 들면 다음과 같이 전략적 설계를 적용할 수 있다.

### 🔸 하위 도메인 구분
| 하위 도메인 | 설명 | 대표 컨텍스트 |
|-------------|------|----------------|
| Core | 수익과 직결되는 핵심 가치 | `Sales`, `Pricing` |
| Supporting | 핵심을 지원하지만 차별화 요소는 아님 | `Catalog`, `Inventory` |
| Generic | 표준 솔루션 사용 가능 | `Billing`, `Identity` |

### 🔸 바운디드 컨텍스트 정의 예
- `Sales BC`: 견적, 주문 흐름 담당. `Order`, `Quotation` 등 핵심 용어 사용.
- `Inventory BC`: 재고 보유량, 입출고 기록. `Stock`, `Reservation` 등 별도 용어 사용.
- `Billing BC`: 외부 PG사 연동 및 세금 계산. `Invoice`, `Settlement` 용어 사용.

### 🔸 컨텍스트 맵 관계
- `Sales ↔ Inventory`: Partnership (양측이 재고 정책을 맞추기 위해 긴밀 협력)
- `Sales → Billing`: Anti-Corruption Layer (Sales 모델을 지키기 위해 송장 모델을 변환)
- `Sales ↔ Pricing`: Shared Kernel (가격 계산 규칙을 코드 레벨로 공유)

## 6. 예제 코드
아래는 Java 기반으로 `Sales` 컨텍스트가 `Billing` 컨텍스트와 통신할 때 Anti-Corruption Layer를 활용해 자체 모델을 보호하는 예시다. 각각은 독립 배포가 가능한 모듈/서비스라고 가정한다.

```java
// sales-context/src/main/java/com/example/sales/domain/Order.java
public class Order {
    private final String id;
    private final String customerId;
    private final Money total;

    public Order(String id, String customerId, Money total) {
        this.id = id;
        this.customerId = customerId;
        this.total = total;
    }

    public String getId() {
        return id;
    }

    public Money totalAmount() {
        return total;
    }
}

// sales-context/src/main/java/com/example/sales/application/PlaceOrderService.java
public class PlaceOrderService {
    private final OrderRepository orderRepository;
    private final BillingGateway billingGateway;

    public PlaceOrderService(OrderRepository orderRepository, BillingGateway billingGateway) {
        this.orderRepository = orderRepository;
        this.billingGateway = billingGateway;
    }

    public void execute(PlaceOrderCommand command) {
        Order order = new Order(command.orderId(), command.customerId(), command.amount());
        orderRepository.save(order);

        // Anti-Corruption Layer를 통해 외부 Billing 모델로 변환
        billingGateway.requestInvoice(order);
    }
}

record PlaceOrderCommand(String orderId, String customerId, Money amount) {}

// sales-context/src/main/java/com/example/sales/infra/BillingGateway.java
public class BillingGateway {
    private final BillingClient billingClient;

    public BillingGateway(BillingClient billingClient) {
        this.billingClient = billingClient;
    }

    public void requestInvoice(Order order) {
        BillingInvoiceRequest dto = BillingInvoiceRequest.fromSales(order);
        billingClient.sendInvoiceRequest(dto);
    }
}

// sales-context/src/main/java/com/example/sales/infra/BillingClient.java
public interface BillingClient {
    void sendInvoiceRequest(BillingInvoiceRequest request);
}

// billing-acl/src/main/java/com/example/billingacl/BillingInvoiceRequest.java
public final class BillingInvoiceRequest {
    private final String externalOrderCode;
    private final BigDecimal priceInWon;

    private BillingInvoiceRequest(String externalOrderCode, BigDecimal priceInWon) {
        this.externalOrderCode = externalOrderCode;
        this.priceInWon = priceInWon;
    }

    public static BillingInvoiceRequest fromSales(Order order) {
        Money total = order.totalAmount();
        return new BillingInvoiceRequest(
            order.getId(),
            total.amount()
        );
    }
}

// billing-context/src/main/java/com/example/billing/BillingService.java
public class BillingService {
    public void issueInvoice(BillingInvoiceRequest request) {
        // Billing 컨텍스트 고유 용어(Invoice, Settlement 등)로 비즈니스 규칙 적용
    }
}
```

위 코드에서 `sales-context`는 자신의 용어(`Order`, `Money`)를 유지한 채 ACL(`BillingInvoiceRequest`)을 통해 `Billing` 컨텍스트가 이해할 수 있는 DTO로 변환한다. 이를 통해 컨텍스트 간 모델 충돌을 피하면서 명확한 협력 관계를 유지할 수 있다.
