# Saga / Process Manager

## 1. 한 줄 정의
**Saga**는 분산 트랜잭션을 하나로 묶는 대신,  
여러 로컬 트랜잭션과 보상(Compensation)으로 전체 일관성을 맞추는 방식이다.

**Process Manager**는 그 흐름을 상태 기반으로 조율하는 컴포넌트다.

---

## 2. 왜 필요한가?
MSA에서 여러 서비스에 걸친 작업을 한 번에 처리해야 할 때가 많다.

예: 주문 생성 -> 결제 승인 -> 재고 차감 -> 배송 시작

이 과정을 하나의 DB 트랜잭션처럼 묶기 어렵기 때문에  
실패 시 복구 전략(보상 트랜잭션)이 필요하다.

---

## 3. 핵심 개념
1. **Local Transaction**: 각 서비스 내부 트랜잭션
2. **Compensation**: 실패 시 되돌리는 반대 동작
3. **Orchestration**: 중앙 조율자가 순서를 제어
4. **Choreography**: 이벤트 기반으로 각 서비스가 자율 반응

---

## 4. 구조 선택
### Orchestration (중앙 조율)
- 장점: 흐름 가시성이 좋고 복잡한 분기 처리에 유리
- 단점: 조율자 복잡도가 증가

### Choreography (분산 자율)
- 장점: 서비스 자율성, 느슨한 결합
- 단점: 전체 흐름 파악/디버깅 난이도 증가

---

## 5. 예시 시나리오 (주문-결제-재고)
### 정상 흐름
1. `OrderCreated`
2. `PaymentApproved`
3. `StockReserved`
4. `OrderConfirmed`

### 실패 흐름
1. `OrderCreated`
2. `PaymentApproved`
3. `StockReserveFailed`
4. 보상: `PaymentCancelRequested`
5. 보상 완료: `OrderCancelled`

핵심은 "어디서 실패해도 비즈니스적으로 허용 가능한 상태"로 마무리하는 것이다.

---

## 6. 간단 상태 모델
| 상태 | 설명 | 다음 상태 |
|------|------|-----------|
| PENDING | 프로세스 시작 | PAYMENT_DONE / FAILED |
| PAYMENT_DONE | 결제 성공 | STOCK_DONE / COMPENSATING |
| STOCK_DONE | 재고 성공 | COMPLETED |
| COMPENSATING | 보상 실행 중 | CANCELLED / FAILED |
| COMPLETED | 정상 종료 | - |
| CANCELLED | 보상 종료 | - |

---

## 7. 장점과 단점
### 장점
- 2PC 없이 분산 프로세스 처리 가능
- 실패 복구 경로를 명시적으로 설계 가능
- 장기 실행 비즈니스 플로우에 적합

### 단점
- 상태 관리와 모니터링 복잡도 증가
- 보상 로직 설계가 어려움
- 재시도/중복 처리 정책이 필수

---

## 8. 자주 하는 실수
### 실수 1: 보상을 "완전 롤백"으로 오해
현실에서는 완전 복구가 불가능할 수 있다.  
비즈니스적으로 허용 가능한 복구 상태를 정의해야 한다.

### 실수 2: 타임아웃/재시도 정책 미정의
네트워크 지연/부분 실패가 기본 전제다.

### 실수 3: 추적 ID 없이 운영
분산 흐름은 `sagaId`, `correlationId` 없으면 장애 분석이 매우 어렵다.

---

## 9. 최소 체크리스트
- 각 단계의 성공/실패/보상 이벤트가 정의되어 있는가?
- 보상 트랜잭션이 실제로 실행 가능한가?
- 타임아웃/재시도/중복 처리 규칙이 있는가?
- 상태 전이 로그와 모니터링 대시보드가 있는가?

---

## 10. 요약
Saga/Process Manager는  
**분산 환경에서 실패를 전제로 일관성을 관리하는 방식**이다.

핵심은 "실패하지 않는 시스템"이 아니라,  
"실패해도 복구 가능한 프로세스"를 만드는 것이다.
